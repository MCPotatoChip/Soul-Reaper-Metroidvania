# Bleach: Soul Society Underground — Codebase Overview

## Summary
This is a single-file 2D Metroidvania action-platformer built in vanilla JavaScript rendered on an HTML5 Canvas. The entire game lives in one file: `artifacts/bleach-game/public/game.js` (~7000+ lines). It features Ichigo Kurosaki exploring Soul Society and Hueco Mundo, unlocking abilities (wall jump, dash, Getsuga Tenshō, Cero, Bankai, Hollow Mask), fighting Hollows and Arrancar enemies, and defeating four major bosses (Renji, Byakuya, Grimmjow, Ulquiorra). The game is part of a monorepo that also contains a React UI shell and an API backend — but the actual game is a standalone canvas application that runs directly in the browser with no framework dependencies.

## Architecture
- **Primary pattern**: Single-file procedural architecture. Everything — constants, player state, enemy AI, boss AI, NPCs, room data, rendering, audio, UI, input handling, and the game loop — exists in one file.
- **Game states**: Title → Play → Boss Intro / Dialogue / Shop / Map / Dead / Ability / Settings. The `gs` variable (enum-like object `GS`) drives which branch of the game loop executes.
- **No framework**: The game uses raw Canvas 2D API. Pixel-art sprites are drawn procedurally using `fillRect` calls. No external game libraries or physics engines.
- **Audio**: Uses Web Audio API for procedural sound effects (oscillator-based) and generative music tracks (arpeggios). No audio files are loaded.
- **Monorepo context**: The game sits inside `artifacts/bleach-game/` which is a Vite + React + Tailwind project, but the game itself has zero React dependencies — it runs as a `<script>` in `index.html`.
- **Entry point**: `gameLoop()` is the single `requestAnimationFrame` loop. It calls `IN.poll()`, handles state-specific logic, and then calls `render()`.

## Key Abstractions

### Player (`P`)
- **File**: `game.js` (line ~1100)
- **Responsibility**: Holds all player state — position, velocity, HP, Reiatsu, coins, unlocked abilities, animation state, Bankai/Hollow timers. Contains the `update()`, `collide()`, `takeDmg()`, `die()`, `respawn()`, and `draw()` methods.
- **Key properties**: `mHP/hp` (max/current HP), `mSl/sl` (max/current Reiatsu), `hasDash/hasWJ/hasGet/hasCero/hasDJ/hasBankai/hasHollow` (ability flags), `bk/bkActive` (Bankai meter and active state), `hm/hmActive` (Hollow meter and active state), `charging/chargeT` (manual Reiatsu charge), `aT/aD` (attack timer/direction), `dT/dD` (dash timer/direction), `inv` (invincibility frames).
- **Lifecycle**: Created at game start (`P.x=128; P.y=400`), reset on respawn. Unlockable abilities are permanently set to `true` by NPC dialogue.

### Boss (`Bs`)
- **File**: `game.js` (line ~1400)
- **Responsibility**: Boss enemy lifecycle — states (`idle`, `dash`, `jump`, `teleport`, attack states), phase transitions at 50% HP, health bar rendering, projectile spawning.
- **Key types**: `renji` (HP 60), `byakuya` (HP 90), `grimmjow` (HP 120), `ulquiorra` (HP 150). Each has phase 2 at 50% HP with enhanced attack patterns.
- **Activation**: Bosses activate when the player enters within 200px range, triggering `BOSS_INTRO` game state with timed cutscene (180 frames).

### Room Data (`RM`)
- **File**: `game.js` (line ~400)
- **Responsibility**: Static tile maps, enemy spawn points, NPC positions, transitions, chest placements, boss definitions. Each room is a plain object with `zone`, `w`, `h`, `tiles` (2D array generated by IIFEs), `enemies`, `npcs`, `boss`, `trans`, `chests`, `save`.
- **Design pattern**: Tile arrays are generated via inline IIFEs that programmatically place platforms, walls, and obstacles.

### Input Abstraction (`IN`)
- **File**: `game.js` (line ~80)
- **Responsibility**: Unified input system supporting keyboard, gamepad, and mobile touch. Each frame `poll()` merges all three input sources into `_held`/`_pressed` arrays. Bindings are configurable and saved to `localStorage`.
- **Actions**: 15 actions (`left`, `right`, `up`, `down`, `jump`, `attack`, `getsuga`, `cero`, `dash`, `heal`, `charge`, `interact`, `map`, `bankai`, `hollow`) each with keyboard keys and gamepad button indices.

### Enemy (`En`)
- **File**: `game.js` (line ~1250)
- **Responsibility**: Generic enemy class. Different behaviors driven by `type` field: `hollow` (ground-based with charge attacks), `fly` (aerial with swoop attacks), `guard`/`arrancar` (ground combatants with sword attacks), `menos` (large, fires Cero), `adjuchas` (fast ground predator with jump attacks). Each has its own AI method (`upHollow`, `upFly`, `upGuard`, `upMenos`, `upAdjuchas`).
- **Key mechanics**: `canWalk()` wall+edge detection, `resFull()` collision resolution, `takeDmg()` with knockback and slow-timer support (from Frozen Blade weapon mod).

### Room Transitions
- **File**: `game.js` — `doTrans()` function and `RM[room].trans` arrays
- **Responsibility**: Seamless room-to-room movement. Each room defines transition zones (x-coordinate, y-range, destination room, spawn coordinates). Transitions have a 350ms cooldown (`tLock`) to prevent rapid re-triggering.

### Weapon Modifiers
- **File**: `game.js` (~line 1075)
- **Key data**: `weaponMods` object with `frozenBlade` (slows enemies, ice particles) and `whipBlade` (+55% melee range). Purchasable from shop. Effects are checked throughout combat, projectile, and rendering code.

## Data Flow
1. **Input** → `IN.poll()` merges keyboard + gamepad + touch into unified `_held`/`_pressed` state
2. **Game Loop** → `gameLoop()` checks `gs` (game state), branches accordingly
3. **Update Phase** (gs===PLAY): `P.update(room)` → physics + ability handling → `P.collide(room)` → tile collision resolution
4. **Combat**: `combat()` checks player attack hitbox vs enemy bounds → applies damage → increments combo counter
5. **Enemy AI**: Each enemy's `update(room)` runs behavior FSM (patrol/chase/telegraph/attack) → `resFull(room)` for collision
6. **Boss**: Boss `update(room)` runs state machine → spawns boss projectiles (`bPr`) → player takes contact damage
7. **Projectiles**: `upPr(room)` and `upBP(room)` filter projectile arrays, check tile/enemy/player collisions
8. **NPC Check**: `checkNPC()` checks proximity + interact key → opens dialogue/shop/heal
9. **Render Phase**: Draw backgrounds → tiles → save point → chests → NPCs → enemies → boss → boss projectiles → player → player projectiles → particles → fog → floating damage numbers → HUD/boss intro overlay
10. **Cleanup**: `IN.clear()` resets per-frame pressed states

## Non-Obvious Behaviors & Design Decisions

### The entire game is one file
This is a deliberate choice for simplicity — a Replit-hosted project meant to be easily sharable. All game logic, rendering, and data lives in `game.js`. Any changes must be made in that single file.

### Hollow mask is NOT drawn on Ichigo during Hollowfication
The `hmActive` flag sets a white/red aura and boosts damage by 1.25x, but **no hollow mask is drawn on Ichigo's face**. The sprite drawing code (`drIchigo`) only draws a partial mask when the NPC context passes `hm=true`. The player sprite rendering call passes `this.hmActive` but the mask drawing code only triggers in the sprite function context — looking closely at `drIchigo()`, the mask block IS drawn when `hm` parameter is true. However, the rendering block draws a mask that covers only the left portion (x+2 to x+9). This appears functional but the mask may not be prominent enough visually.

### Bosses can be hit before activation
Bosses exist as entities immediately on room load but only activate when the player enters within 200px. The `combat()` function checks boss hitboxes regardless of `boss.active` status. This means the player can attack (and potentially defeat) a boss before the cutscene even triggers. **Fix needed**: Add `boss.active` check to the combat attack hitbox test.

### Enemy glitch behavior — root causes identified

**Yoruichi level (r3)**:
- Enemies: `{type:'hollow',x:22,y:12}`, `{type:'fly',x:35,y:8}`, `{type:'hollow',x:38,y:13}`
- Issue: Enemies near solid tiles (the fly at x:35 is near a pit, the hollow at x:38 near the right wall). The `canWalk()` method and `resFull()` collision resolution can cause enemies to clip when knockback or AI reversal pushes them into solid tiles.
- Fix: Move enemy spawns further from walls/pits and add a safety margin in `resFull()`.

**Seireitei gate (s1)**:
- Guards at `{x:5,y:13}`, `{x:14,y:10}`, `{x:28,y:13}`
- Player enters at x:2 — guard at x:14 is within ~12 tiles. The guard at the door (x:5) is near the left wall boundary.
- Fix: Move the x:14 guard to x:18+ and x:5 guard to x:8+.

**Inner Seireitei (s2)**:
- Guards at `{x:4,y:13}`, `{x:22,y:13}`, `{x:35,y:13}`, fly at `{x:15,y:4}`
- Guard at x:4 is 2 tiles from entrance at x:2. Guards near column walls at x:10, x:20, x:29 can glitch when patrol AI reverses against solid tiles.
- Fix: Move entrance guard to x:8+, reposition guards away from wall columns.

**Rukia level (r2)**:
- Rukia NPC at `{x:17,y:12}`, enemies at `{x:9,y:12}` and `{x:22,y:10}`
- Player enters from r1 at x:17 — nearly on top of Rukia. Enemies are within 5-7 tiles of the NPC.
- Fix: Move Rukia to x:5-6, move enemies further away from both entrance and NPC.

**Nel level (h2)**:
- Nel NPC at `{x:4,y:13}`, save point (Orihime) also at approx `{x:4,y:13}` in tile coordinates
- Save point is at `save:{x:4,y:13}` which is the same tile coordinate as Nel NPC
- Fix: Separate Nel and the save point by at least 6-8 tiles.

**Hueco Mundo Desert (h1)**:
- The `adjuchas` enemy (x:22,y:9) matches the "dog that hits 2 hits" description — it has `dmg:2`, a wolf/dog sprite with spine bumps and tail, and its AI (`upAdjuchas`) includes jump attacks (`this.vy=-8.5` or `-5.5` during telegraph→attack transitions).
- The adjuchas already has jump attacks in its AI.

### Reiatsu passive regen and active charge
- **Passive regen**: Line in `P.update()`: `if(!this.charging&&fr%180===0&&this.sl<this.mSl)this.sl=Math.min(this.mSl,this.sl+1);` — every 180 frames (~3 seconds at 60fps), +1 Reiatsu passively.
- **Active charge**: Holding C key charges Reiatsu at `0.35 + 0.25` per frame when `chargeT > 60` (roughly 0.6/frame ≈ 36/sec at 60fps).
- **To remove both**: Delete the passive regen line AND the entire charging block.

### L-attack (Getsuga Tenshō / Cero) range and damage
- **Getsuga**: `dmg: Math.floor(15 * pwrMult)`, lifetime 40 frames, speed `d*8` (where d is 12 normally, 8 in bankai=8*12=96px/sec). Range = 40 frames × 96px/sec ≈ 384px (~12 tiles).
- **Cero**: `dmg: Math.floor(12 * pwrMult)`, lifetime 60 frames, speed 14 pixels/frame. Range = 60 × 14 = 840px (~26 tiles).
- **Change requested**: Getsuga damage from 15→8, Cero damage from 12→8, reduce range (reduce lifetime from 40→30 for Getsuga, from 60→35 for Cero).

### Chest in r3 (Yoruichi level) — accessible from underneath
- **Chest at**: `{id:'r3_c1', x:41*T, y:9*T}` — this is in the upper alcove.
- The alcove has floor platforms at `t[10][41]=2, t[10][42]=2, t[10][43]=2` (tile row 10).
- Below the alcove there could be open space. The player on row 11-14 tiles could potentially reach the chest by jumping up through platforms (since platform tiles `type===2` are one-way).
- **Fix**: Add solid blocks (tile type 1) beneath the alcove chest area so it's only accessible from the sides via wall jump or from above after breaking the breakable wall (type 5) at `t[7][40], t[8][40], t[9][40]`.

### Bankai and Hollow scream audio
- The game currently uses Web Audio API procedural sounds. The `AU.play()` system plays synthesizer-generated sounds.
- Bankai currently plays `AU.play('ability')` (a sine wave sweep). Hollow mask plays `AU.play('hollow')` (a square wave effect).
- **To add "BANKAI!" scream and hollow scream from YouTube**: You would need to:
  1. Download audio files (MP3/WAV)
  2. Add them to `public/` directory
  3. Create an `<audio>` element approach or use `AudioContext.decodeAudioData()` to load and play them
  4. Trigger them in the Bankai and Hollow activation code instead of the procedural sounds

### Boss cutscene invincibility
- Currently `BOSS_INTRO` state lasts 180 frames. During this time `gs !== GS.PLAY` so `update()` doesn't run, meaning `combat()` is NOT called. However, the boss intro state only triggers after `boss.activate()` is called (which sets `gs=GS.BOSS_INTRO`). The boss is invulnerable during the cutscene because the game loop is in the BOSS_INTRO branch which only calls `render()`.
- However, if the player attacks the boss BEFORE entering the 200px activation range (by shooting Getsuga from far away), they can damage the boss pre-activation since `combat()` checks boss hitboxes without checking `boss.active`.
- **Fix**: In `combat()`, change the boss attack check from `if(boss&&boss.hp>0&&rO(ab,boss.bounds()))` to `if(boss&&boss.hp>0&&boss.active&&rO(ab,boss.bounds()))`.

### Bleach-themed UI changes
- The current HUD uses simple rectangles and monospace fonts. To make it "Bleach-like" with "retro screen and bleach-themed HP":
  - **HP**: Currently green circles. Could be changed to use a stylized "substitute soul reaper badge" motif or red/black skull symbols
  - **Reiatsu bar**: Could use blue flame/spirit ribbon styling
  - **Font**: Use a Japanese-calligraphy-inspired font (loaded via CSS `@font-face` or Google Fonts)
  - **Retro screen effect**: Add CRT scanlines overlay, slight color bleeding, and a vignette
  - **Bankai/Hollow gauges**: Could use stylized flame/eye designs
  - All changes are in `dHUD()` function in game.js and related CSS in index.html
